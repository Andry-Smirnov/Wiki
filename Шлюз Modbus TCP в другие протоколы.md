# Шлюз Modbus TCP в другие протоколы

Шлюз позволяет выполнять запросы на подключенные к контроллеру устройства посредством Modbus TCP, адресуя подключенные устройства с помощью `Unit ID` заголовка MBAP (стр.23²).

## Сервер Modbus TCP

При получении Modbus TCP-запроса с `Unit ID` = `255` (или `0`) шлюз обрабатывает этот запрос как запрос к его внутренним регистрам, при этом операции записи выполняются в начале цикла, а операции чтения – в конце, что позволяет в одну транзакцию выполнять запись регистров и сразу получать результат выполнения команд в отклике (см. команду `0x17 Read/Write Holding Registers`), а так же иметь более актуальное значение данных по сравнению с вариантом формирования отклика на команду чтения в начале цикла.

## Шлюз Modbus TCP

При получении Modbus TCP-запроса c `Unit ID` ≠ `255`, `0` или `1` сервер перебирает имеющихся ведущих сетей и как только найдётся ведущий, которому принадлежит ведомый с пришедшим `Unit ID`, такой запрос помещается в очередь запросов соответствующего ведущего, при этом ведущий запоминает `IP:Port` и `Transaction ID` запроса для возврата отклика клиенту, а команды записи (Write Coils/Registers) помещаются в очередь с более высоким приоритетом исполнения. Если ни один из ведущих не признает `Unit ID` как адрес одного из подключенных к нему ведомых, то сервер вернёт клиенту exception `10 GATEWAY PATH UNAVAILABLE`.

Благодаря наличию очереди реализованный шлюз допускает формирование TCP-запросов на один и тот же `Unit ID` не дожидаясь получения отклика, однако если ведущему не удастся выделить память для размещения запроса клиенту вернётся exception `6 SERVER DEVICE BUSY`, что говорит о необходимости снизить частоту запросов от клиентов.

Каждый ведущий чередует вызов своих ведомых для формирования запросов технологической программы и запросы из очереди, подменяя в последнем случае `Unit ID` на `Slave ID`*.

При получении отклика на запрос записи из очереди (команды Write Coils/Registers) ведущий удаляет выполненную команду из очереди и возвращает полученный отклик на ранее запомненный `IP:Port` с ранее запомненным Transaction ID, в случае получения любого отклика на запрос чтения из очереди (или не из очереди, если в очереди нет команд записи на откликнувшегося абонента) ведущий сразу возвращает отклик всем абонентам, выполнившим такой же запрос чтения (Read Coils/Input/Registers). Поскольку у ведущего помимо обработки запросов, полученных по Modbus TCP, могут быть запросы, обусловленные технологической программой, следует иметь в виду, что отклик на запросы может быть затянут по времени, кроме того, очерёдность получения откликов через шлюз в общем случае не соответствует очерёдности запросов.

Если на запрос ведущего Modbus RTU-ведомый не откликнется или будет получен отклик с ошибкой – клиент получит exception `11 GATEWAY TARGET DEVICE FAILED TO RESPOND` вместо ожидаемого отклика.

Сервер разрывает Modbus TCP соединение в следующих случаях:

1. В ответ на разрыв соединения клиентом.
2. При наличии четвёртого Modbus TCP-соединения с одного и того же IP-адреса (отключается соединение, с которого давно не поступало запросов).
3. При получении некорректного MBAP (Transaction ID не меняется, Protocol ID ≠ 0, значение поля Length менее двух или более 253х).

В остальных случаях (даже при невалидном запросе) соединение не рвётся, а запрос может быть проигнорирован.

## Шлюз Modbus RTU over TCP

Реализованный шлюз позволяет иметь одно Modbus RTU over TCP соединение для ретрансляции запросов фирменного ПО подключенных устройств.

Алгоритм обработки запросов этого соединения несколько отличается от Modbus TCP – поскольку отсутствует однозначная возможность сопоставить полученный отклик с отправленным запросом, очередь не содержит более одного RTU-запроса на один `Unit ID` и появление нового запроса автоматически удаляет из очереди все предыдущие RTU-запросы на тот же `Unit ID` (не затрагивая TCP-соединения) – предполагается, что если клиент отправил запрос не дождавшись – значит он считает, что таймаут ожидания отклика истёк. Такое поведение делает бессмысленным приоритет команд записи, поскольку очередь всегда содержит только последнюю команду на заданного `Unit ID`.

Сервер разрывает соединение Modbus RTU over TCP в следующих случаях:

1. В ответ на разрыв соединения клиентом.
2. При подключении нового клиента по Modbus RTU over TCP с любого IP-адреса.
3. При некорректном фрейме запроса (например, не сошлась контрольная сумма).

В случае, если фирменное ПО начинает работать со сбоями, обрывая соединение или не дожидаясь откликов, рекомендуется увеличить его таймаут ожидания отклика.

## Сноски

- Для простоты обслуживания рекомендуется на линиях с одним ведущим для `Slave ID` выбирать значение по умолчанию, устанавливаемое производителем, а на линиях, где предусмотрено более одного ведущего – стараться не иметь на линии устройства с умолчательным `Slave ID`, чтобы в случае замены оборудования не было коллизий – самое простое – выбирать `Slave ID` равным `Unit ID` – в принципе этому правилу можно следовать и на линиях с одним ведомым.
