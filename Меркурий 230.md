# Меркурий 230

Протокол не имеет ничего общего с Modbus RTU, кроме сетевого адреса в первом байте запроса/отклика и способа расчёта контрольной суммы и её расположения.

По аналогии с кодом функции Modbus RTU и не имея к нему никакого отношения запросы содержат во втором байте код запроса.

Сетевой адрес прибора может принимать значения от `1` до `240`, адрес `254` предназначен для широковещательных запросов, а адрес `0` используется для запросов к единственному абоненту на линии (однако использовать этот адрес для команд управления или изменения параметров нельзя).

Отклик на команды записи или при ошибках возвращает 1 байт состояния обмена, младшая тетрада которого содержит код ошибки:

| Младшая тетрада | Описание |
| --- | --- |
| `0` | всё нормально |
| `1` | недопустимая команда или параметр |
| `2` | внутренняя ошибка счётчика (ошибка при обмене с памятью №2 для 3.32 изменение режима разъединителя, стр.35) |
| `3` | недостаточен уровень доступа для выполнения запроса |
| `4` | внутренние часы счётчика уже корректировались в течение текущих суток |
| `5` | не открыт канал связи |
| `6` | самовольно задействован по аналогии с модбасовским exception `6 SERVER DEVICE BUSY` |

Для получения данных со счётчика требуется авторизация командой открытия канала связи (2.2.2, стр.10¹), без неё счётчик можно только пинговать (но это неточно) командой тестирования канала связи (2.1, стр.9¹).

## Детали реализации

Modbus TCP-запросы клиента передаются на ведомого как есть (с заменой MBAP на `Slave ID` и добавлением в конец контрольной суммы), отклики от ведомого передаются клиенту как есть (с обратной заменой Slave ID на MBAP и удалением контрольной суммы).

Реализован режим с единственным абонентом, при включении контроллера запросы к Меркурию идут со `Slave ID` = `0`, который изменяется на реальный при получении отклика на команду записи (3.6 стр.21¹) или чтения (4.4.7 стр.65¹) сетевого адреса, при этом для команды чтения сетевого адреса по Modbus TCP over RTU (выбор типа интерфейса TCP/IP в конфигураторе²) возвращаемое значение подменяется на `Unit ID` (иначе конфигуратор при инициализации связи переключается на полученный адрес, что делает работу конфигуратора при `Slave ID` ≠ `Unit ID` невозможным), т.е. при записи другого значения конфигуратор продолжит видеть `Unit ID`, хотя оно фактически изменилось. Чтение сетевого адреса по Modbus TCP такого побочного эффекта не имеет и возвращает фактическое значение `Slave ID`.

Кроме того, шлюз блокирует выполнение команд записи параметров встроенного интерфейса (3.13 стр.24) и параметров основного интерфейса (3.14.3 стр.26), возвращая клиенту байт состояния обмена с кодом `1` – недопустимая команда или параметр, поскольку их изменение приведёт к потере связи с прибором.

## Умолчательные параметры связи

9600 8n1, доступные значения – 300×2ⁿ (где n=0…7), 57600, 115200, 8e1, 8o1
